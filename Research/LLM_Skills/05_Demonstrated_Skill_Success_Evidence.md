# 00_ADMIN/Research/LLM_Skills/05_Demonstrated_Skill_Success_Evidence.md

## Scope, method, and inclusion criteria

This session asked a narrower question than a general game-development tooling survey: whether **reusable LLM skills** or **comparable reusable agent instructions/workflows** measurably improve software-engineering or adjacent development outcomes. In this environment, I could not directly open the five local project files named in the prompt, so the project-specific framing below is limited to the prompt text itself rather than the inaccessible local markdown files. The evidence review therefore focused on public primary sources that were accessible at run time: official documentation, official repositories, benchmark papers, benchmark leaderboards, issues, and engineering retrospectives.

I treated **skills** in the OpenAI sense as reusable workflows made of instructions plus optional scripts/resources, and I treated adjacent evidence as relevant when the reusable element was an instruction scaffold, agent-computer interface, review workflow, or stable prompt protocol that can be reused across tasks or repositories. That operationalization matches current OpenAI documentation, which describes skills as reusable workflows and the authoring format for reusable instructions, resources, and scripts. citeturn42search0turn42search1turn42search4turn42search7turn42search12

Inclusion favored evidence that had at least one of the following: a meaningful baseline, a reproducible benchmark, repeated production use with measurable outcomes, public artifacts such as issue histories or workflow logs, or explicit negative results. I excluded popularity, GitHub stars, “awesome” lists, and long prompt collections as evidence of success by themselves. I also downgraded any case where the claimed benefit could more plausibly be explained by a stronger model, a better interface, larger context, memory, more tools, or closer human supervision than by the reusable instructions themselves. That downgrade matters because several sources explicitly show that tool/interface changes or memory systems can dominate the effect size of prompt-like changes. citeturn22view1turn36view0turn38view0

The search set included: official OpenAI Codex skills documentation and the official `openai/skills` and `openai/openai-agents-python` repositories; SWE-bench-related academic papers and leaderboards; GitHub engineering posts with offline benchmarks or A/B tests; and public issue histories for actual skill failures. OpenAI’s own guidance also recommends evaluating skills systematically with evals, which is methodologically aligned with the inclusion criteria used here. citeturn42search5turn32search14turn29view0turn26search0turn26search1

## Strongest evidence cases

### SWE-agent shows that a reusable interface-and-instructions scaffold can materially outperform weaker baselines

**What was used.** SWE-agent is not a `SKILL.md` package, but it is squarely in scope as a reusable agent scaffold: a fixed **agent-computer interface** plus associated instructions/workflow for repository navigation, search, editing, and test execution. The paper’s core claim is that interface design for the agent matters, and the benchmarked scaffold is reusable across tasks. citeturn21view0turn22view1

**Model, tools, environment.** The main reported configuration used GPT-4 Turbo as the base LM, with an ACI built on top of the Linux shell that exposes LM-friendly search, file-view, and editing commands plus guardrails and concise feedback. The paper also reports portability to Claude 3 Opus. citeturn21view0turn22view1turn22view4

**Task and baseline.** The benchmark task was real GitHub issue resolution on SWE-bench and SWE-bench Lite. The most relevant baselines were a non-interactive RAG baseline and a shell-only agent. On full SWE-bench, SWE-agent with GPT-4 Turbo reached **12.47%** resolved, substantially above the earlier **3.8%** non-interactive RAG best cited by the paper. On SWE-bench Lite, the shell-only baseline reached **11.00%** while SWE-agent reached **18.00%**, a **10.7 percentage-point** improvement attributable to the ACI and associated workflow rather than a model change. On HumanEvalFix, SWE-agent with GPT-4 Turbo reached **87.7/89.7/87.9** pass@1 across Python/JS/Java, far above the non-interactive models listed in the same table. citeturn22view1turn22view2turn22view4turn23view0turn23view1

**What changed and how success was measured.** The paper’s ablations are especially important because they isolate reusable workflow/interface choices. On SWE-bench Lite, the same basic agent family performed worse when editing/search were less specialized, and the ablation table shows the tuned design outperforming alternatives in editor, search, viewer, and context-management choices. That is stronger causal evidence than “we wrote a nice prompt and it seemed better.” citeturn23view0turn22view0

**Reproducibility, failures, and confounders.** This is **Strong** evidence because the paper openly emphasizes reproducibility, releases code and artifacts, and provides benchmarked ablations with a meaningful baseline. At the same time, the causal story is not “instructions alone”: the gains are inseparable from the custom interface and feedback design. The right conclusion is therefore narrower and more useful: **reusable, task-shaped scaffolds measurably help when they narrow actions, sharpen feedback, and prevent common errors**. That is highly relevant to skill design, but it also warns against attributing everything to prompt text. citeturn21view0turn22view1turn23view0

**Rating.** **Strong**.  
**Transferability to this project.** High for repository navigation, code editing, verification, and diff-focused review workflows; lower for game feel, art direction, or broad multi-hour design work without a strong verification harness.

### Agentless is the strongest counter-hype result in the space

**What was used.** Agentless uses a reusable two-phase workflow—**hierarchical localization followed by repair**—and explicitly disallows autonomous planning and complex tool usage. That makes it directly relevant to the question because it demonstrates that a simpler, reusable instruction workflow can outperform many more elaborate “agent” stacks. citeturn18view0turn24view1

**Model, tools, environment.** The reported open-source headline used GPT-4o, with the workflow centered on hierarchical localization into files, then classes/functions/variables, then edit locations, followed by repair. The method is intentionally simple and low-tool. citeturn19view0turn24view1

**Task and baseline.** The task was SWE-bench Lite issue resolution. Against existing open-source agent systems, Agentless solved **82/300 = 27.33%** at **$0.34** average cost, which the paper reports as the highest open-source performance and lowest cost among the compared open-source approaches in that evaluation. The paper also gives a compact ablation: single sample **23.33%**, multiple samples plus majority voting **26.00%**, full Agentless with test filtering **27.33%**. citeturn19view0turn24view0

**What changed and how success was measured.** The workflow improvement was measured through resolved-task rate on SWE-bench Lite plus cost. This is important because it is not a popularity story: the paper argues that the simple, interpretable workflow beat many more complex open-source agents on the same benchmark. citeturn18view0turn19view0turn24view0

**Reproducibility, failures, and confounders.** This is **Strong** evidence, but with a different lesson than many skill advocates want. The result does **not** show that “more skills” or “richer prompts” necessarily help. It shows that a reusable workflow can help when it aggressively constrains the problem and reduces context. It also provides negative evidence about benchmark quality: the authors found SWE-bench Lite problems with exact ground-truth patches in the description (**4.3%**), insufficient information (**9.3%**), and misleading solutions (**4.3%**), then created **SWE-bench Lite-** to filter those out. That caveat matters because any claimed skill success on this benchmark inherits some benchmark noise. citeturn24view4turn24view3

**Rating.** **Strong**.  
**Transferability to this project.** High for bug fixing, localized refactors, regression debugging, and scope control. Very relevant for avoiding over-agentic workflows that wander, over-read, or invent extra files.

### AutoCodeRover supports structured retrieval-and-repair workflows, with measurable gains over simpler baselines

**What was used.** AutoCodeRover is another reusable scaffold rather than a `SKILL.md`, but its contribution is again a reusable workflow: issue solving with structured AST-aware search, iterative context retrieval, and optional test-based spectrum-based fault localization. citeturn18view1

**Model, tools, environment.** The paper describes an LLM-based issue resolver that uses class/method-aware search APIs and, when available, spectrum-based fault localization from tests. citeturn18view1turn19view3

**Task and baseline.** On SWE-bench Lite, AutoCodeRover @1 resolved **19.00% (57 tasks)** at **37k tokens / $0.43**, compared with SWE-agent’s **18.00% (54 tasks)** and **245k tokens / $2.51** in the cited comparison table. AutoCodeRover @3 rose to **26.00% (78 tasks)**, and the SBFL-enhanced variant reached **22.00% (66 tasks)**. citeturn19view2

**What changed and how success was measured.** The main measured gains came from structured retrieval and, in one ablation, additional fault-localization signals. The paper also claims that about **two-thirds** of the autonomously produced patches they examined were correct and acceptable, which is notable because raw “resolved” counts can mask patch acceptability. citeturn18view1

**Reproducibility, failures, and confounders.** This is **Strong** evidence for structured workflow value, but again the causal driver is not prompt wording alone. The likely causal ingredients are AST-aware search, sharper context control, and test-based localization. For a skills program, the transferable lesson is that **reusable instructions pay off most when they enforce a structured retrieval discipline and verification loop**. citeturn18view1turn19view2

**Rating.** **Strong**.  
**Transferability to this project.** Medium to high for engine/API debugging, test-guided bug fixing, and configuration/integration work; lower for creative systems design without executable checks.

### Self-Debugging is strong adjacent evidence, but it is more about execution loops than repository engineering

**What was used.** The reusable element is a debugging workflow: generate code, inspect execution results, explain the code, and iteratively self-debug. That is adjacent rather than identical to repository-level software engineering, but still relevant because it is a reusable instruction protocol with benchmarked gains. citeturn19view5

**Model, tools, environment.** The paper focuses on code-generation settings, not repository issue resolution. The workflow relies on execution feedback or code explanation rather than repository navigation. citeturn19view5

**Task and baseline.** The paper reports state-of-the-art gains on Spider, TransCoder, and MBPP. The load-bearing result is that self-debugging improved baseline accuracy by **2–3%** on Spider, improved hardest Spider problems by **9%**, and improved baseline accuracy by **up to 12%** on TransCoder and MBPP. It also improved sample efficiency enough to match or beat baselines that generated more than **10x** as many candidate programs. citeturn19view5

**Negative evidence.** A later paper, **Revisit Self-Debugging with Self-Generated Tests**, is the critical caveat: it found that **post-execution** self-debugging with self-generated tests “struggles on basic problems” and that gains depended on setting and design; only **in-execution** self-debugging showed a more robust improvement pattern in their study. That directly warns against simplistic “always write tests and self-debug” skill recipes. citeturn19view4

**Rating.** **Strong** for adjacent code-generation workflows; **Moderate** for direct transfer to repository-scale software engineering.  
**Transferability to this project.** Medium for standalone scripts, gameplay utility code, data transforms, and test-driven bug isolation; weaker for broad cross-file architectural tasks.

## Moderate evidence cases

### GitHub’s Copilot code review rewrite is one of the best public before-and-after cases for reusable review instructions

GitHub publicly described a concrete regression-and-fix cycle in Copilot code review. After migrating to a shared `grep`/`glob`/`view` toolset, offline benchmarks showed **higher review cost** and **fewer useful issues caught**. GitHub then rewrote the workflow instructions so the agent behaved like a reviewer—starting from the diff, narrowing with search, batching discovery, and reading only exact evidence—and reported that the regression flipped into a production win: **roughly 20% lower average review cost while maintaining the same review quality**. GitHub also reports a useful negative case: the same focused instructions did **not** produce the same win in Copilot CLI, because the CLI’s job is broader and more exploratory. That is unusually good public evidence that reusable, task-shaped instructions can change outcome and cost, but also that the gain is **job-specific rather than universal**. citeturn38view0

Against the ten requested questions: **workflow used** was diff-first review guidance around shared tools; **model/toolset** was Copilot code review using GitHub’s shared CLI-style tools; **task** was pull-request review; **baseline** was the same shared tools under worse guidance; **change** was the instructions, not the tool layer; **success** was measured in review cost and review-quality signal; **reproducibility** is limited because the benchmark is internal; **negative case** is explicitly reported for Copilot CLI; **major confounder** is that the tools were also changed, though GitHub argues the regression/fix isolated instructions; **transferability** is high for diff review, low for open-ended coding. On the rubric here, that makes it **Moderate**, not **Strong**. citeturn38view0

### GitHub’s cross-agent memory A/B tests show measurable gains, but those gains are not skill gains

GitHub’s cross-agent memory system is not a skill system, yet it is a vital confounder because it shows that **repository-scoped validated memory** can produce measurable engineering outcomes. In their evaluation, repository memories with citation verification produced a **3% precision increase** and **4% recall increase** for code review in the evaluation set, and A/B tests on live developers showed a **7% increase in pull-request merge rates** for Copilot coding agent (**90% with memories vs. 83% without**) and a **2% increase in positive feedback on review comments** (**77% vs. 75%**), with reported **p < 0.00001**. citeturn36view0

This matters because many “skills worked” stories may actually be memory or state-management stories. The measurable benefit here comes from persistent, repository-scoped knowledge with real-time citation verification, not from a static reusable instruction file. For this review, that makes the case **Moderate** evidence that agent guidance layers can help engineering outcomes, but also **strong confounder evidence** against over-crediting skills alone. citeturn36view0

### GitHub Agentic Workflows provides moderate evidence that workflow restructuring can reduce token use without changing behavior

GitHub’s agentic-workflows engineering post is another public before/after case. They instrumented production workflows, built daily audit/optimizer workflows, then removed unused MCP tools and substituted deterministic GitHub CLI steps for always-needed fetches. In smoke tests, pruning unused MCP tools cut per-call context by **8–12 KB** and saved several thousand tokens per run with **no change in behavior**. Across production workflows, GitHub reported effective-token-spend reductions such as **62%** for Auto-Triage Issues over **109 post-fix runs**, **19%** for Daily Compiler Quality over **12** post-fix runs, **37%** for Daily Community Attribution over **8** post-fix runs, and **43–59%** for two frequently run security/integration workflows. citeturn40view0

This is relevant because many reusable “skills” are really workflow policies: move deterministic reads out of the LLM loop; restrict tools; pre-stage mandatory evidence; instrument cost. The caution is that most of the gain here comes from **workflow and tooling refactoring**, not from better natural-language instructions. It is therefore **Moderate** evidence for reusable workflow guidance, but weak evidence for skill text alone. citeturn40view0

### OpenAI’s `openai-agents-python` repository shows repeated real-world skill use, but not public controlled outcome measurements

OpenAI’s `openai-agents-python` repository is the clearest official public example I found of skills used for live repository maintenance. The root `AGENTS.md` does not just mention skills; it mandates them for certain classes of work. It requires `$code-change-verification` before completing changes that affect runtime code/tests/build behavior, requires `$implementation-strategy` before user-facing runtime changes, and requires `$pr-draft-summary` before final responses after eligible code work. citeturn29view0

The inspected skills are concrete and repository-shaped rather than generic prompt bundles. `$code-change-verification` runs a mandatory verification stack and its scripts invoke formatting, linting, type checking, and tests. `docs-sync` scopes documentation audits against `main` or the diff versus `main`. `final-release-review` audits the diff from the latest release tag to the release candidate for regressions and breaking changes. `pr-draft-summary` standardizes PR summaries and branch suggestions. The repository also contains `runtime-behavior-probe` references that explicitly teach recurring validation patterns, including control runs, parity checks, and environment false-signal handling. OpenAI’s own examples show Codex/sandbox tasks being instructed to use named skills such as `$credit-note-fixer`, and the repository’s action logs show a flow of PRs such as “docs: sync examples.md with current examples directory,” which is at least consistent with repeated use of skills in repository operations. citeturn30view0turn30view1turn30view2turn30view3turn30view4turn28search9turn28search12turn28search2turn28search5

What is missing is the key thing this session asked for: **public measurable before/after outcomes for those specific skills**. The repository proves real use and active maintenance; it does not publicly prove defect reduction, faster merges, fewer regressions, or lower token cost for those specific skills. That makes the evidence **Moderate** for “skills are used seriously in a real engineering repository” and only **Weak-to-Moderate** for “these specific skills measurably improved engineering outcomes.” citeturn29view0turn30view0turn30view1turn30view2turn30view3turn30view4

## Weak, promotional, and speculative claims that are often overstated

OpenAI’s current skills documentation is clear about the intended value proposition: skills are reusable workflows, they can package instructions plus resources/scripts, and they are meant for repeatable work across ChatGPT desktop, Codex CLI, and related surfaces. The official `openai/skills` repository similarly presents skills as discoverable folders of instructions, scripts, and resources that agents can use for specific tasks, and emphasizes installability through `$skill-installer`. Those are important product definitions, but they are **not by themselves evidence of success** in software engineering. citeturn42search0turn42search1turn42search4turn42search7turn32search14turn33search17

OpenAI’s “Testing Agent Skills Systematically with Evals” blog is methodologically useful because it states that skill names/descriptions are the main trigger signals and that evals are necessary. But that source is chiefly guidance about evaluation practice, not published evidence that a particular engineering skill materially improved real software outcomes. It should therefore be treated as **process guidance**, not as proof of skill efficacy. citeturn42search5

The same caution applies to official documentation from other vendors. GitHub says agent skills are folders of instructions, scripts, and resources, and Microsoft’s agent-skills docs similarly describe portable packages following an open specification. Anthropic’s help center says skills improve consistency, speed, and performance through progressive disclosure. Those statements are credible product descriptions, but in this session’s rubric they are still **Promotional** unless they are paired with a baseline, artifact trail, or reproducible evaluation. citeturn7search22turn11search13turn42search2

Public issue histories in the official `openai/skills` repository also show why curated skill catalogs should not be assumed to work well just because they are official. There are documented problems such as `gh-address-comments` failing to find its helper script and falling back to GraphQL, `skill-creator` choosing the wrong directory, `gh-address-comments` breaking on fork PRs, and an open issue about malformed YAML generation from `skill-creator`. These are not reasons to reject the whole pattern; they are reasons to reject any claim that official or popular skills are automatically robust. In this rubric, such cases are **Weak** evidence of practical friction and **negative evidence** against naive adoption. citeturn34search1turn33search8turn33search12turn33search11

## Causal confounder analysis

The strongest confounder is **model and interface quality**. SWE-agent’s gains are tied to a custom interface with guardrails and concise feedback, not merely a better prompt. AutoCodeRover’s gains depend heavily on structured code search and optional fault localization. GitHub’s code review case shows that the same tools under different instructions can change cost and quality, but the setup still depends on the tool surface itself. Any claim that “skills improve engineering outcomes” needs to be narrowed to “skills improve outcomes when paired with a task-shaped action space and good feedback.” citeturn22view1turn18view1turn38view0

A second confounder is **persistent memory and context management**. GitHub’s memory A/B tests show live improvements in merge rates and review feedback from repository-scoped memory with citation verification. If a team adds both skills and memory, the measured lift may come mostly from memory, not from the skill file. For The Bible Video Game project, this matters because stable repository facts—engine version, scene organization, save format, build pipeline, art import conventions—may be better handled as durable validated memory or repository references than as ever-growing skill text. citeturn36view0

A third confounder is **benchmark construction**. Agentless explicitly found benchmark-quality problems inside SWE-bench Lite and built a filtered subset. That does not invalidate the positive results, but it should discourage over-precise claims about rank order differences between methods. For a project-facing decision, the safer takeaway is directional: constrained localization, diff-first review, and verification loops help more reliably than giant instruction bundles. citeturn24view4

A fourth confounder is **human supervision and environment shaping**. GitHub’s token-efficiency gains came from engineering the workflow around deterministic CLI prefetching and audit jobs, not from trusting the model to “figure it out.” Likewise, OpenAI’s repository skills are attractive partly because they encode human repository policy—run the checks, compare against latest release tag, use control runs, write the PR summary—not because the model became intrinsically better. These are still genuine improvements, but they are improvements in **process discipline** as much as in model behavior. citeturn40view0turn29view0turn30view0turn30view2turn30view4

The most important causal lesson for this project is therefore conservative: **small, verifiable, repository-shaped workflows have the best evidence; broad “AI-native team brain” skill bundles do not**. The evidence supports localized reusable workflows that reduce ambiguity, constrain scope, stage evidence, and force verification. It does not support assuming that a large skill catalog, by itself, will make a game-development agent reliable. citeturn22view1turn24view0turn38view0turn40view0

## Transferability analysis for The Bible Video Game

The best-supported transferable pattern is the **narrow, high-discipline engineering skill**. For a game project, that means workflows such as “code change verification,” “scene or prefab review against a diff,” “release-readiness review,” “runtime behavior probe,” or “docs/code sync for exported APIs and build settings.” These are analogous to the strongest evidence cases because they are bounded, have clear baselines, and can be checked with tests or direct repository evidence. OpenAI’s own repository skills are written in exactly that style, and the SWE-agent / Agentless / AutoCodeRover papers all reward workflows that narrow the search space and couple action to verification. citeturn29view0turn30view0turn30view1turn30view2turn30view4turn22view1turn24view0turn19view2

The second most transferable pattern is **diff-first review guidance**. Game projects are unusually exposed to scope creep, surprise asset churn, and “fixed the bug by rewriting half the subsystem” behavior. GitHub’s code-review case is highly relevant because the winning instruction pattern started from the diff, asked narrow questions, and minimized unnecessary repository exploration. For a game project, that translates into review skills that explicitly ask: what scene/prefab/script changed, what nearby systems are coupled, what tests or play-mode checks are mandatory, and what evidence is enough to stop searching. That kind of workflow is especially valuable when avoiding unrequested features or unnecessary files is a priority. citeturn38view0

The third transferable pattern is **verification before completion**. OpenAI’s `$code-change-verification` skill is mundane, but that is precisely why it is valuable: formatting, linting, typing, and tests are run before work is marked complete. In a game repository, the analogous pattern would adapt to engine reality: compile/build check, targeted unit or play-mode tests where available, scene-load validation, asset-import sanity checks, and platform-specific smoke checks. The evidence base here is not controlled in the academic sense, but it aligns with the highest-signal real-world repository practice I found. citeturn29view0turn30view0

What transfers poorly is the idea of a **large general-purpose skill catalog** for game development. The current public evidence does not show that broad catalogs improve architectural consistency, visual quality, gameplay feel, or multi-engine portability. The official docs support the concept, but the measurable wins are almost all from **small, task-specific workflows**. For game-dev-adjacent visual validation loops, I did not find strong public evidence comparable to the software-engineering benchmark cases above. That is a research gap, not proof of failure. citeturn42search0turn42search1turn42search18

The practical scope consequence is straightforward. If The Bible Video Game wants evidence-backed reuse, the safest leverage points are **build/test/review/release/debugging workflows** rather than broad creative or design skills. That does not diminish the project’s ambition; it simply marks which parts of the ambition are backed by public evidence today, and which parts still require experimentation or internal evals. citeturn22view1turn24view0turn38view0turn40view0

### Research gaps

The largest gap is that I did **not** find public, independently reproducible studies showing that **repository-local `SKILL.md` files themselves**, separate from interfaces, memory, or tooling, measurably reduce defects or regressions in software engineering. The public evidence is markedly stronger for reusable **scaffolds/workflows** than for reusable **skill files** as a distinct causal layer. citeturn22view1turn24view0turn38view0

A second gap is game-development-specific evidence. I did not find strong public controlled studies on reusable skills for Unity, Godot, Unreal, Blender, or game-specific visual validation loops that match the rigor of SWE-bench-style software-engineering evidence. The absence of evidence here should be treated as a research gap, not as evidence that skills do not help. citeturn42search13

A third gap is public reporting on **negative cases**. The strongest public papers include some failure or caveat analysis, but official product and repository posts still rarely quantify regressions, false-positive rates, unnecessary file creation, scope creep, or “fixed the benchmark, broke the repo” behavior. GitHub’s July 2026 code-review post is unusually valuable precisely because it reported a regression before reporting the fix. More of that kind of reporting is needed. citeturn38view0

A fourth gap is independent replication of official engineering case studies. GitHub’s memory and code-review posts are more informative than typical vendor marketing because they include baselines and metrics, but they are still primarily first-party reports. Independent eval packs for “instruction-file changes in the same agent/tool/model stack” would materially improve confidence. citeturn36view0turn38view0
