## Verified storage and discovery model

Current official OpenAI documentation says a local Codex skill is a directory containing a required `SKILL.md` plus optional `scripts/`, `references/`, `assets/`, and `agents/openai.yaml`. Discovery starts from metadata rather than the full instructions: Codex begins with each skillŌĆÖs `name`, `description`, and file path, and only reads the full `SKILL.md` if it decides to use the skill. References are read and scripts are run only when needed. That progressive-disclosure model is stated directly in the skills guide and reiterated in OpenAIŌĆÖs OSS-maintenance blog post. ŅłĆciteŅłéturn40view0Ņłéturn39view1Ņłü

For **local Codex discovery**, the currently documented locations are: repository scope under `.agents/skills` in every directory from the current working directory up to the repository root, user scope under `$HOME/.agents/skills`, admin scope under `/etc/codex/skills`, and built-in system skills bundled with Codex. The current guide also explicitly says Codex supports **symlinked skill folders** and follows the symlink target when scanning those locations. This is solid enough to treat as the current official discovery model. ŅłĆciteŅłéturn40view0Ņłü

For **repository instructions**, Codex separately builds an `AGENTS.md` instruction chain once per run or session start. It reads one global file from `~/.codex`, then walks from project root down to the current directory, preferring `AGENTS.override.md`, then `AGENTS.md`, then configured fallback names in each directory. Files closer to the current directory override broader files because they appear later in the combined prompt, and the combined size is capped by `project_doc_max_bytes`, which defaults to 32 KiB. ŅłĆciteŅłéturn41view0Ņłéturn32view1Ņłéturn32view2Ņłü

For **API-hosted skills**, the model is different. The Skills API requires a bundle with exactly one top-level `SKILL.md` or `skill.md`, validates it, versions it on the OpenAI side, and then mounts it into hosted shell via `tools[].environment.skills`. The same guide explicitly says hosted shell supports uploaded `skill_reference` attachments and explicit versions, while local shell does not; local shell expects local files in the runtime you control. That means local Codex skill discovery and API-hosted skill attachment should be treated as adjacent but separate implementation targets. ŅłĆciteŅłéturn13view0Ņłéturn36view0Ņłéturn36view5Ņłü

Official repository evidence also shows that OpenAIŌĆÖs current reusable-distribution center of gravity has shifted from **`openai/skills` to `openai/plugins`**. The public `openai/skills` README is marked **deprecated** and points readers to `openai/plugins`, while `openai/plugins` documents plugin bundles as the current installable packaging unit for skills, connectors, MCP servers, and related assets. For current projects, `openai/skills` is historical evidence, not current canon. ŅłĆciteŅłéturn5view2Ņłéturn17view2Ņłü

### Skill package anatomy

The currently documented **required** anatomy for a local Codex skill is minimal: a directory plus `SKILL.md`, and the `SKILL.md` must contain at least `name` and `description` in frontmatter. The API-side Skills guide adds that frontmatter validation follows the Agent Skills specification and that exactly one `skill.md`/`SKILL.md` is allowed in an uploaded bundle. That minimum package shape is stable enough to standardize on. ŅłĆciteŅłéturn40view0Ņłéturn13view0Ņłü

OpenAIŌĆÖs Codex docs add an OpenAI-specific optional file, `agents/openai.yaml`, which can carry UI metadata, invocation policy, and tool dependencies. The documented fields currently include `interface.display_name`, `short_description`, icons, `brand_color`, and `default_prompt`; `policy.allow_implicit_invocation`; and `dependencies.tools` entries such as MCP dependencies. This file should be treated as **Codex-specific optional enhancement metadata**, not as part of the portable minimum skill anatomy. ŅłĆciteŅłéturn40view0Ņłü

Public official repository examples confirm that the docs reflect real packaging, not just theory. The `openai/plugins` README says each plugin has a required `.codex-plugin/plugin.json` and may include `skills/`, `.app.json`, `.mcp.json`, plugin-level `agents/`, `commands/`, `hooks.json`, `assets/`, and other supporting files. Current public plugin manifests in that repo show practical use of `name`, `version`, `description`, `author`, `homepage`, `repository`, `license`, `keywords`, `skills`, `apps`, `mcpServers`, and install-surface `interface` metadata. ŅłĆciteŅłéturn17view2Ņłéturn15view1Ņłéturn15view2Ņłéturn15view3Ņłü

The same repository also confirms the **progressive-disclosure supporting-file pattern**. The current Figma plugin includes multiple skills, individual `SKILL.md` files, `references/` documents, and executable `scripts/`. Public raw files show the frontmatter and body of `figma-design-to-code`, a substantial `error-recovery.md` reference document for a multi-call design-system workflow, and a `cleanupOrphans.js` helper script. That is strong evidence that supporting references and scripts are part of real current Codex skill ecosystems, not merely optional placeholders. ## Invocation, lifecycle, and context loading

OpenAI currently documents **two official invocation paths** for local Codex skills: explicit and implicit. Explicit invocation means naming the skill directly in the prompt; in CLI and IDE, the documented interfaces are `/skills` and `$` mention. Implicit invocation means Codex chooses a skill when the task matches the skillŌĆÖs `description`. Because of that, the docs repeatedly frame the description as routing metadata rather than prose decoration. ŅłĆciteŅłéturn40view0Ņłéturn24search3Ņłü

The OpenAI API guide describes the same routing idea for hosted shell. The platform adds each skillŌĆÖs `name`, `description`, and `path` to **user-prompt context**, the model decides whether to invoke a skill from that metadata, and it reads the full `SKILL.md` only if it chooses to use it. The same guide explicitly says skill instructions are **user prompt input, not system prompt input**. That is one of the most important currently documented precedence facts. ŅłĆciteŅłéturn13view0Ņłéturn10view5Ņłü

For **explicit local activation**, the current App Server docs are more precise than the surface UI docs. A client can send `$<skill-name>` in text and, recommendedly, add a `skill` input item carrying the skill name and path. If the `skill` item is omitted, the model will still parse `$<skill-name>` and try to locate the skill, but the docs warn that this can add latency. The command docs also say the `/skills` picker inserts selected skill context so the **next request** follows that skillŌĆÖs instructions. ŅłĆciteŅłéturn25view0Ņłéturn29search10Ņłü

The **context-loading budget** is unusually concrete in current official docs. Codex says the initial visible skills list uses at most **2% of the model context window, or 8,000 characters when the context window is unknown**. If many skills are installed, Codex shortens descriptions first; if the set is still too large, it may omit some skills from the initial list and show a warning. The full `SKILL.md` still loads when a skill is actually selected. That is stable enough to matter for design decisions around description length and catalog size. ŅłĆciteŅłéturn40view0Ņłü

### What is documented versus unresolved about lifecycle

What is **officially documented** is that `AGENTS.md` is built once per run or TUI session start, explicit `/skills` affects the next request, and skill bodies are loaded on demand rather than globally. What is **not clearly documented** is whether an implicitly selected skill remains sticky across follow-up turns, whether an explicitly selected skill persists for a full session, and exactly when previously loaded skill instructions are discarded or compacted in long-running Codex threads. Because OpenAI does not currently specify that lifecycle precisely, that behavior should be treated as **unverified for project policy purposes**. ŅłĆciteŅłéturn41view0Ņłéturn29search10Ņłéturn40view0Ņłü

Multiple-skill behavior is only partly documented. Official docs clearly say a **plugin can bundle multiple skills**, and product materials show multiple skills and plugins appearing inline in the composer. App Server input also supports explicit `skill` items inside a list-shaped turn payload. But OpenAI does **not** currently provide a crisp normative statement that multiple skills compose with deterministic semantics when present together in one turn. That makes ŌĆ£simultaneous multi-skill compositionŌĆØ an **observed possibility, not a fully specified contract**. ŅłĆciteŅłéturn40view0Ņłéturn26search1Ņłéturn25view0Ņłü

## Precedence, conflict behavior, and composition

`AGENTS.md` precedence is clearly documented. Global guidance comes first, then project guidance from root to leaf, with nearer directories overriding broader ones because they are appended later. Within a directory, `AGENTS.override.md` wins over `AGENTS.md`, and Codex rebuilds the chain on every run or TUI session start. This is stable enough to encode directly into repository governance. ŅłĆciteŅłéturn41view0Ņłéturn28view0Ņłü

Skill precedence is **not comparably well documented**. The current skills guide says that if two skills share the same `name`, Codex **doesnŌĆÖt merge them** and **both can appear in skill selectors**. That is the clearest current official statement. But a public issue in the official `openai/codex` repo reports duplicate same-name skills in the CLI `$` selector and describes the expected behavior as if a higher-precedence skill should win. Because that expectation is **not** stated in the current official docsŌĆöand in fact sits awkwardly beside the ŌĆ£both can appearŌĆØ languageŌĆöthe deterministic winner for same-name collisions should be treated as **unresolved**. ŅłĆciteŅłéturn40view0Ņłéturn35view2Ņłü

The relationship between **repository instructions** and **skill instructions** is also only partly inferable. The AGENTS docs and Codex Prompting Guide say AGENTS files are injected as **user-role messages** near the top of the conversation before the user prompt, while the API Skills guide says skill instructions are **user prompt input** rather than system input. What OpenAI does **not** currently state is a direct rule like ŌĆ£skills override AGENTSŌĆØ or ŌĆ£AGENTS override skills.ŌĆØ The safest current inference is that they sit in the same broad user-instruction layer, with ordering likely mattering, but the exact conflict rule is **not officially specified**. ŅłĆciteŅłéturn41view0Ņłéturn28view0Ņłéturn13view0Ņłü

Composition through **declared dependencies** is better documented than instruction precedence. `agents/openai.yaml` can declare `dependencies.tools`, including MCP endpoints. OpenAIŌĆÖs customization docs explicitly describe **skills + MCP** as the intended composition point: skills define the repeatable workflow, while MCP provides external tools and context. Plugins then package one or more skills with connector mappings and MCP server configuration for installation and sharing. That is stable enough to encode into project skills. ŅłĆciteŅłéturn40view0Ņłéturn26search5Ņłéturn31search0Ņłü

### Behavior when a capability is unavailable

OpenAI is much clearer about environment and permission failures than about routing failures. Local Codex runs commands in a sandbox; `git`, package managers, linters, and test runners inherit the same sandbox boundaries; network is off by default; and destructive or side-effecting app or MCP calls can require approval. If a reviewer or user denies a boundary-crossing action, Codex is instructed to find a materially safer path or stop and ask. That is stable platform behavior. ŅłĆciteŅłéturn12view0Ņłéturn12view1Ņłéturn12view7Ņłü

What is **not** standardized at the skill layer is how a skill should narrate missing dependencies, missing auth, or absent capabilities. OpenAIŌĆÖs own and third-party public skill files often include instructions to report the exact failure rather than silently invent a fallback, but that pattern is visible mainly in repository examples and blog advice, not in a formal skills-spec rule. For project authoring, that means this is a **recommended pattern**, not a guaranteed Codex runtime behavior. ŅłĆciteŅłéturn23search3Ņłéturn39view0Ņłéturn39view2Ņłü

## Tools, shell, sandbox, MCP, Git, and portability

The OpenAI API Shell guide says skills are ŌĆ£reusable, versioned bundlesŌĆØ that can be mounted in hosted shell environments, and at shell execution time the **model decides whether to invoke them**. The guide also distinguishes hosted shell from local shell: hosted containers are managed by OpenAI, while local shell runs in a runtime you or the user control. That is the cleanest official statement of the relationship between skills and execution: a skill describes a workflow, and shell executes the concrete commands when the model chooses that route. ŅłĆciteŅłéturn10view4Ņłéturn36view4Ņłü

On local Codex surfaces, sandboxing is foundational. The sandbox applies to spawned commands, not only built-in file operations, so `git`, test runners, package managers, and other CLI tools work inside the same boundaries. By default, network access is off, workspace-write mode limits writable areas, and approvals mediate attempts to cross those boundaries. Destructive app or MCP actions always require approval when tools advertise destructive annotations. This means skill authors should assume that tool use succeeds only when the environment already permits it. ŅłĆciteŅłéturn12view0Ņłéturn12view1Ņłéturn12view2Ņłü

MCP is documented as a first-class extension route. ChatGPT web can use remote MCP-backed tools supplied by plugins, while local Codex clients can connect directly to MCP servers and share that configuration between CLI and IDE. The Codex developer-command reference documents `codex mcp` and `codex mcp-server`, and the Docs MCP quickstart says the CLI and IDE share MCP configuration. That is stable enough to make MCP the preferred integration mechanism when a skill depends on structured external tools. ŅłĆciteŅłéturn12view3Ņłéturn12view5Ņłéturn12view6Ņłü

Git integration is also first-class rather than incidental. Codex-managed worktrees live under `$CODEX_HOME/worktrees`, start from the selected branch head, and can copy ignored setup files into new worktrees via `.worktreeinclude`. Any skill that assumes a particular checkout shape, ignored local files, or mutable branch state needs to account for worktrees explicitly if it is meant to be reliable. ŅłĆciteŅłéturn12view4Ņłéturn30search16Ņłü

One subtle but important current-platform detail is that `.agents` and `.codex` directories are called out as protected read-only paths in current Codex documentation. That means skills that attempt to self-modify installation roots, rewrite config under those directories, or mutate other installed skills may hit protection boundaries. For project engineering, it is safer to treat those locations as configuration and distribution surfaces, not normal writable workspace directories. ŅłĆciteŅłéturn38search1Ņłü

## Versioning, distribution, security, licensing, and provenance

For **local Codex skills**, OpenAI documents discovery, enable/disable controls, and plugin packaging, but it does **not** document a first-class standalone versioning scheme for a raw `.agents/skills/<skill>` directory. By contrast, **plugins** have a manifest-level `version`, and App Server documents plugin marketplace sources as local, Git, npm package-registry, or remote catalog entries. That makes plugins the current official reusable distribution unit, while direct skill folders remain the local authoring and repo-scoped mechanism. ŅłĆciteŅłéturn40view0Ņłéturn25view0Ņłéturn31search0Ņłü

For **API skills**, versioning is explicit and first-class. The Skills guide exposes `default_version`, `latest_version`, and `skill_reference.version`, and the cookbook clarifies that each upload creates one skill bundle with one top-level `SKILL.md`, while multiple skills require multiple bundles. Any hosted-environment adoption should use that API version model rather than inventing a parallel folder-based version scheme. ŅłĆciteŅłéturn13view0Ņłéturn36view5Ņłü

Security guidance is strong and should be treated as mandatory. The API Skills guide warns that skills can introduce prompt-injection-driven data-exfiltration risks, especially with network access. Local Codex security docs add that network access is off by default, commands run in a sandbox, and app or MCP actions with side effects can require approval. Marketplace access can also be restricted through allowed-source requirements. This is stable enough to encode into project policy: third-party skills should be treated as executable operational content, not inert markdown. ŅłĆciteŅłéturn13view0Ņłéturn12view0Ņłéturn12view1Ņłéturn32view3Ņłü

Licensing and attribution are plugin-level concerns today, not afterthoughts. The plugin build docs say plugin manifests commonly include `author`, `homepage`, `repository`, `license`, and `keywords`, and public manifests in `openai/plugins` show that licenses vary by plugin: some are MIT, while others use product-specific license identifiers such as `LicenseRef-Figma-Developer-Terms`. A repository that adapts external skills should therefore preserve attribution metadata and review the upstream license on a per-source basis rather than assuming all shared skills are permissively reusable. ŅłĆciteŅłéturn31search0Ņłéturn15view1Ņłéturn15view2Ņłéturn15view3Ņłü

For provenance, the safest current official pattern is the one OpenAI describes in its own OSS-maintenance post: keep repository-specific workflows in repo-local `.agents/skills`, use `AGENTS.md` to declare when those workflows are mandatory, and package cross-repository distribution as plugins. That same post explicitly describes using repo-local skills, `AGENTS.md`, optional scripts and references, and CI reuse together in the OpenAI Agents SDK repos. That is stable enough to use as the projectŌĆÖs default operating assumption. ŅłĆciteŅłéturn39view0Ņłéturn39view1Ņłü

## Limitations, unresolved questions, and claim-status table

Several load-bearing questions remain unresolved or contradictory in current official material. The most important are the exact precedence rule for **same-name skill collisions**, the exact conflict rule between **AGENTS instructions and skill instructions**, the persistence window for a selected skill across follow-up turns, and current **IDE plugin support**. There is also a small but real inconsistency in the official App Server page, which says it reads `interface` and `dependencies` from `SKILL.json`, while the main skills guide documents `agents/openai.yaml`. That should be treated as a documentation gap unless and until OpenAI resolves it explicitly. ŅłĆciteŅłéturn25view0Ņłéturn40view0Ņłü

There are also clear signs of historical churn in community and maintainer discussion. Early-2026 maintainer comments and user issues refer to `.codex`-based skill paths, experimental installer flows from `openai/skills`, and discovery regressions that no longer match todayŌĆÖs canonical `.agents/skills` guidance. Those sources are useful as evidence that behavior changed over time and that edge cases exist, but they are **not** safe to treat as universal present-tense Codex behavior.
