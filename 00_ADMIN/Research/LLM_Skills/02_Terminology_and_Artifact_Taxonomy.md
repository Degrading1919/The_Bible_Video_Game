This report uses **finding** for evidence-backed description and **recommendation** for a proposed project choice. A named artifact is not assumed to possess authority merely because an agent can load it. Likewise, an artifact that describes an action is not itself capable of performing that action.

## 1. Executive definition summary

The central distinction is between **instructions**, **actors/capabilities**, **knowledge**, **state transfer**, and **evidence**:

- Instructions—including system, developer, repository, skill, prompt, workflow, policy, and role text—shape behavior but do not act independently.
- Agents may reason and select actions; tools, scripts, hooks, and some MCP-server operations can cause effects, but only within granted permissions and runtime controls.
- Domain references and context files carry knowledge. Loading them does not make their claims true or elevate them above Scripture, approved project decisions, or source criticism.
- Memory and handoffs carry state across time or context boundaries. They are fallible summaries, not self-authenticating authority.
- Evaluations and benchmarks supply evidence about behavior. They do not create safety by existing; they must be run, scored, reviewed, and maintained.

In current Codex terminology, a **skill** is a reusable, focused package consisting of a `SKILL.md` plus optional references, assets, and scripts. Codex can load it explicitly or when its description matches a task; this builds on the open [Agent Skills specification](https://agentskills.io/specification). A **custom agent** is a configured agent type with its own instructions and potentially different model, tool, MCP, or sandbox settings; a **subagent** is a delegated agent instance working on a bounded task. `AGENTS.md` is persistent repository guidance discovered by Codex, not an agent definition and not a skill. MCP is a protocol for connecting clients to servers that expose capabilities such as tools, resources, and prompts; an MCP server is therefore not itself a skill or a single tool. These distinctions are described in OpenAI's current documentation for [skills](https://learn.chatgpt.com/docs/build-skills), [`AGENTS.md`](https://learn.chatgpt.com/docs/agent-configuration/agents-md), [subagents](https://learn.chatgpt.com/docs/agent-configuration/subagents), and [MCP](https://learn.chatgpt.com/docs/extend/mcp).

For this project, the existing Master Assistant Protocol is a governing project-policy and role document; the Activation Prompts are manual role-launch prompts; and the Continuity Handoff Prompt is a state-transfer template. None is evidence of an implemented skill, router, agent configuration, memory system, or evaluation harness. That finding from [Session 1](https://github.com/Degrading1919/The_Bible_Video_Game/blob/e121e44c209d496ad0a5ac5b61bd402fb5e2bd62/00_ADMIN/Research/LLM_Skills/01_Project_Context_and_Authority_Map.md) remains controlling for repository-state claims.

## 2. Canonical terminology table

“Cargo” below means what an artifact primarily carries: **K** knowledge, **R** role/decision ownership, **P** procedure, **Y** policy/constraints, **T** tool access/capability, **S** state, or **E** evaluation evidence.

| Term | Operational definition | Authority and lifecycle/scope | Action and cargo | Native status, common confusion, and project relevance |
|---|---|---|---|---|
| **Skill** | A reusable package of focused instructions, with optional references, assets, and scripts, selected for a task. The open format requires a skill directory and `SKILL.md` with `name` and `description`. | Authority comes from the owner who installs/checks it in, subject to higher runtime and project instructions. It persists as a versioned artifact and is activated per task. | The instruction file cannot act; an invoked script or available tool can. Cargo: usually P, sometimes K/Y and tool-use guidance. | Codex-native implementation of the cross-ecosystem [Agent Skills format](https://agentskills.io/specification). Confusing a skill with an agent hides context and permission differences. Candidate for repeatable project work, not an automatic home for every role or rule. |
| **Role** | An allocation of professional lens, responsibilities, decision ownership, inputs, outputs, and prohibited decisions. | Authority comes from the project governance that assigns it. Scope is a task, phase, review gate, or standing organizational function. | Does not act; an agent or person acts while occupying it. Cargo: R/Y and sometimes K expectations. | Generic. Often confused with persona or agent. The nine assistant modes are roles defined by the [Master Protocol](https://github.com/Degrading1919/The_Bible_Video_Game/blob/e121e44c209d496ad0a5ac5b61bd402fb5e2bd62/00_ADMIN/Reference/The_Bible_Video_Game_Master_Assistant_Protocol.md), not nine proven agents or skills. |
| **Persona** | A presentation stance—voice, tone, manner, or simulated identity—used to shape communication. | Authority is stylistic and comes from its assigning instruction. Usually lasts for a turn or session. It should not supersede factual or policy authority. | Cannot act. Cargo: style and behavioral framing, occasionally shallow K. | Generic. Calling a role a persona can reduce real ownership/boundaries to theater; calling a persona an expert can manufacture authority. Reverence is a tone constraint, while Scripture/theology review is a role. |
| **Agent** | A runtime model process that receives instructions/context, reasons over a task, and may select tools in a loop. | Its behavior derives from the active instruction hierarchy, agent configuration, user task, tool grants, and environment. Lifetime is normally a run, chat, or assigned job. | Can initiate permitted actions through tools; the model itself does not bypass tools or permissions. Cargo is supplied rather than inherent. | Generic; exact architecture is vendor-dependent. Confusing an agent with its prompt or config obscures where behavior changed. The repository currently defines assistant roles but no repository-local agent runtime. |
| **Subagent** | A delegated agent instance given a bounded task, normally with a separate context and a result returned to a parent. | Authority is delegated by the parent/user and constrained by inherited and configured instructions/permissions. Lifetime is the delegated job/thread. | Can reason and use its allowed tools. Cargo: task context plus any loaded K/P/Y/T. | Codex-native term and common elsewhere, but inheritance differs by vendor; compare [Codex subagents](https://learn.chatgpt.com/docs/agent-configuration/subagents), [GitHub custom agents](https://docs.github.com/en/copilot/concepts/agents/cloud-agent/about-custom-agents), and [Claude Code subagents](https://code.claude.com/docs/en/sub-agents). A specialist role does not require a subagent. |
| **System instruction** | Highest platform/runtime-level behavioral instruction supplied to the model for a run. | Authority comes from the system operator/platform. Scope is the model request or session; exact composition is implementation-specific. | Does not itself act. Cargo: Y/P, identity, capability boundaries, and context. | Generic industry concept with vendor-specific semantics. It is not a repository policy and cannot be authored by this project in ordinary use. Do not describe the Master Protocol as a system instruction unless actually installed at that layer. |
| **Developer instruction** | Application-owner instruction placed below system authority and above ordinary user task text in OpenAI's instruction model. | Authority comes from the application/deployment owner. Scope is a run/session or configured agent instance. | Does not act. Cargo: Y/P/K and tool-use expectations. | OpenAI-native name; other vendors use different layers. A Markdown role body may become developer instructions when placed in a Codex custom-agent configuration, but is not one merely because it says “must.” |
| **Repository instruction** | Persistent guidance associated with a repository or path and automatically/explicitly loaded by an agent client. | Authority comes from repository governance, bounded by higher platform instructions and current user authority. It lives with the repository and applies by workspace/path. | Does not act. Cargo: Y/P/K, conventions, commands, validation expectations. | Generic class with vendor-specific filenames. It is confused with documentation that is merely present in a repo. The Master Protocol is governing project documentation today, but not yet a verified native repository-instruction artifact. |
| **`AGENTS.md`** | A specifically named Markdown repository-guidance file recognized by Codex and several other coding-agent ecosystems. | Authored by repository/project owners; lifecycle follows the file and its directory scope. Codex composes applicable global/root/nested guidance as documented. | Does not act. Cargo: Y/P/K and routing cues. | Codex-native convention but not OpenAI-exclusive; see [OpenAI](https://learn.chatgpt.com/docs/agent-configuration/agents-md) and [agents.md](https://agents.md). It is neither an agent nor a skill. None exists in the inspected game repository checkout. |
| **Prompt** | Input text or structured message requesting or framing model behavior, from a one-off question to a reusable template. | Authority depends on sender and placement; a user prompt cannot override higher instructions. Lifetime is normally a turn, though conversation history may retain it. | Does not act until interpreted by a runtime. Cargo: requested outcome, K/P/Y, examples, and output shape. | Generic. A prompt is often mislabeled a command, policy, or agent. The Activation Prompts launch roles manually but do not create durable configured agents. |
| **Command** | A discrete invocation recognized by a shell, client, tool, or command registry; some slash commands perform client behavior, while others expand into prompts. | Authority comes from the invoking user/agent plus host permission and command implementation. Scope is one invocation. | May act deterministically, alter client state, or only submit text. Cargo varies. | Generic and highly overloaded. `/skills`, a shell command, and a natural-language imperative are not equivalent. Project build/test commands remain unknown because no engine or toolchain is selected. |
| **Workflow** | An ordered, repeatable sequence of inputs, decisions, actions, checks, and outputs that accomplishes an outcome. | Authority comes from the approved process owner/specification. It persists as a process definition and is instantiated per run. | The description cannot act; an agent, person, script, or automation executes it. Cargo: P/Y and handoff points. | Generic. A workflow may be encoded in a skill, CI file, script, checklist, or orchestration system. The current read→role→work→audit→handoff pattern is manual, not automated. |
| **Policy** | An enduring normative rule stating what is required, permitted, forbidden, or escalated. | Authority comes from a legitimate platform, organization, project, or user decision. It persists until superseded and may apply globally or to a domain/path. | Text cannot act; enforcement mechanisms can block or require approval. Cargo: Y. | Generic. Confusing policy with a skill makes a conditional artifact the sole carrier of an always-on rule. Witness, sacred-event, canon-labeling, and no-invention constraints are project policy. |
| **Guardrail** | A preventative or detective control intended to keep outputs/actions within a boundary; it may be instructional, technical, or review-based. | Authority comes from the policy it enforces. Lifetime follows the control and protected operation. | An instructional guardrail only guides; a permission gate, schema validator, or runtime invariant can constrain action. Cargo: Y/E. | Generic. “Guardrail” does not guarantee enforcement. Sacred-event prose rules are current design guardrails; runtime protections and exploit tests do not yet exist. |
| **Domain reference** | A maintained source of facts, vocabulary, standards, or evidence consulted for subject-matter work. | Authority comes from provenance, source quality, edition/version, and project approval—not location inside a skill. Scope is the domain and source edition. | Does not act. Cargo: K, citations, uncertainty, and definitions. | Generic. Bundling a reference into a skill does not make it canonical. Scripture editions, archaeological research, and regional dossiers require distinct provenance/confidence treatment. |
| **Tool** | A callable capability with defined inputs and outputs that lets an agent read, compute, communicate, or change state. | Availability and authority come from the host/client, provider, permissions, and user approvals. Lifetime is installation/configuration plus each call. | Yes, within its contract and permissions. Cargo: T; results may return K/E/S. | Generic; MCP defines one standardized tool interface. A skill can explain tool use but does not grant the tool. This matters before claiming engine, DCC, repository, or database capability. |
| **MCP server** | A program/service speaking Model Context Protocol and exposing negotiated capabilities such as tools, resources, or prompts to a client. | Authority comes from server operator, client configuration, authentication, and user/tool policy. Lifecycle includes connection/initialization and calls. | The server can provide data and execute exposed operations; effect depends on the called capability. Cargo: T/K/P. | Ecosystem protocol, not Codex-only. The [MCP specification](https://modelcontextprotocol.io/specification/2025-11-25/basic/lifecycle) distinguishes server capabilities, including tools, resources, and prompts. “MCP,” “server,” and “tool” are not synonyms. No project MCP server is selected. |
| **Script** | Executable code that performs a defined operation, usually more deterministic and inspectable than free-form model reasoning. | Authority comes from source ownership, review, execution environment, inputs, and permissions. It persists as code; each run is bounded. | Yes when executed. Cargo: T/P and encoded transformation logic. | Generic. A script bundled with a skill remains separately executable code and must be reviewed. It is preferable for deterministic validation, but no project validation scripts exist yet. |
| **Hook** | Event-triggered automation that runs when a lifecycle event occurs, such as before/after a tool call, edit, commit, or agent step. | Authority comes from host configuration and hook definition. It persists while installed and runs on matching events. | May run scripts, checks, prompts, or notifications; therefore it can act. Cargo: P/Y/T/E. | Generic with vendor-specific events and guarantees; compare Claude Code's [hook reference](https://code.claude.com/docs/en/hooks). A hook is not a workflow merely because it triggers one. Project hooks are premature without a selected development stack. |
| **Memory** | Persisted information made available across turns or sessions, whether user-authored, model-derived, or system-managed. | Authority depends on provenance; persistence and scope are vendor-specific. It lasts beyond the immediate context until revised, expired, or removed. | Does not itself act. Cargo: S/K/preferences. | Generic idea with non-portable implementations. It is often mistaken for truth or a decision log. The project has no implemented AI memory; durable approved decisions should remain inspectable repository artifacts. |
| **Context file** | A file loaded or attached to supply task-relevant facts, constraints, examples, or state. | Authority comes from the file's provenance and its place in project governance. Scope is whichever task/run loads it. | Does not act. Cargo: K/Y/P/S. | Generic. It is a functional use, not necessarily a special format. A research dossier, schema, or handoff may all be context files, but their evidentiary status remains different. |
| **Handoff** | A structured transfer of current state, decisions, evidence, open questions, and next actions between people, agents, or sessions. | Authority comes from cited source decisions and the author’s faithful summary. It lives for a transition and must age visibly. | Does not act. Cargo: S/K/P. | Generic. It is not memory, policy, or a source of truth by itself. The [project handoff prompt](https://github.com/Degrading1919/The_Bible_Video_Game/blob/e121e44c209d496ad0a5ac5b61bd402fb5e2bd62/00_ADMIN/Reference/The_Bible_Video_Game_CONTINUITY_HANDOFF_PROMPT.md) is a template and includes historical assumptions requiring freshness checks. |
| **Evaluation** | A defined procedure and set of cases, expected properties, scoring rules, and review criteria used to measure an agent/artifact. | Authority comes from the evaluation owner and traceable acceptance criteria. Scope is a versioned run/campaign and target behavior. | Criteria do not act; a harness may execute tests and collect E. Cargo: E/Y. | Generic. A checklist is not automatically an evaluation, and a passing score is not theological approval. Future evaluations should test source labeling, bounded agency, schema fidelity, and false-certainty failure cases. |
| **Benchmark** | A standardized, repeatable evaluation dataset/task suite with a scoring method used for comparison across versions or systems. | Authority comes from curators, representativeness, versioning, and scoring integrity. It persists as a named/versioned suite. | Does not itself act; a runner executes it. Cargo: E. | Generic. Every project-specific evaluation need not become a benchmark. Small acceptance suites may be more useful than leaderboard-style comparisons during pre-production. |
| **Agent configuration** | A machine-readable definition that instantiates or modifies an agent’s instructions, model, tools, MCP access, sandbox, or related runtime settings. | Authority comes from the configuration owner plus platform rules. It persists at user/project scope and is applied when an agent/session is created. | The file does not act; it configures an agent that can. Cargo: R/Y/T/P. | Generic class with vendor-specific formats. Codex custom agents use configuration files described in [OpenAI's subagent documentation](https://learn.chatgpt.com/docs/agent-configuration/subagents); GitHub uses agent profiles, and Claude Code uses Markdown/frontmatter. A role description alone is not agent configuration. |

## 3. Codex-native versus external-ecosystem mapping

The names below can look portable while their loading and authority semantics are not.

| Concept | Current Codex realization | External comparison | Portability finding |
|---|---|---|---|
| Reusable skill | `SKILL.md` directory with optional resources/scripts; explicit or description-matched activation. | The [Agent Skills specification](https://agentskills.io/specification) defines the common core; [Claude Code skills](https://code.claude.com/docs/en/slash-commands) add vendor controls. | Core packaging is portable; invocation policy, locations, allowed-tool semantics, and extensions are not guaranteed portable. |
| Persistent project guidance | `AGENTS.md`/override files supply path-scoped instructions. | GitHub Copilot supports several instruction filenames and surfaces, with a [support matrix](https://docs.github.com/en/copilot/reference/custom-instructions-support); Claude uses `CLAUDE.md` and rules. | Natural-language guidance may transfer, but filename discovery, composition, and precedence must be re-audited per client. |
| Specialized agent | Codex custom-agent configuration supplies instructions and optional model/tool/MCP settings. | GitHub [agent profiles](https://docs.github.com/en/copilot/concepts/agents/cloud-agent/about-custom-agents) and Claude [subagent definitions](https://code.claude.com/docs/en/sub-agents) use different schemas and inheritance. | The role concept transfers; configuration and security behavior do not. |
| Delegated worker | Codex subagent/thread returns work to the main agent. | GitHub and Claude also use “subagent,” but isolation, context inheritance, nesting, and tool grants differ. | Treat “subagent” as a conceptual class until vendor/runtime behavior is verified. |
| Reusable prompt/command | Codex custom prompts are documented as deprecated in favor of skills; client and slash commands remain separate concepts. | Claude merged older custom commands into skills; GitHub supports prompt files and CLI commands. | The visible invocation syntax is not a durable architecture. Preserve the underlying workflow, not slash-command branding. |
| Connected capability | Codex can connect to configured MCP servers. | MCP is vendor-neutral at protocol level and defines model-controlled [tools](https://modelcontextprotocol.io/specification/2025-11-25/server/tools), plus resources/prompts. | Protocol messages are portable; authentication, approvals, UI, server trust, and enabled capability policy remain client-specific. |
| Enforcement | Platform/system instructions, sandbox/approvals, and experimental command rules occupy different layers. | Claude permissions/hooks and GitHub tool restrictions use other mechanisms. | A prose “policy skill” is not equivalent to a technical enforcement control on any platform. |
| Memory | Codex products may expose managed memory/context features depending on surface. | Claude supports memory scopes; other agents use files, summaries, or vendor stores. | “Memory” must always name storage, provenance, scope, retention, and editability; otherwise the term is too vague for architecture. |

## 5. Seven-category artifact analysis

These categories are useful analytical labels, but they should not become seven mandatory skill families.

| Proposed category | Finding | Recommendation (not a final catalog) |
|---|---|---|
| **1. Role skills** | Role definitions carry decision ownership and “must not” boundaries. The existing Master Protocol and Activation Prompts already define roles manually. A role can be expressed in a prompt, repository policy, agent configuration, or skill, but a skill’s conditional activation can be unsafe for role boundaries that must always apply. | Keep canonical role ownership in project governance. Consider agent configuration only when a genuinely separate context/tool profile is useful; consider a skill only for the repeatable procedure performed within a role. Do not convert all nine roles automatically. |
| **2. Workflow skills** | A focused, repeated sequence with clear inputs, outputs, decision points, and verification is the strongest default match for a skill. The open specification directly supports instructions plus references and scripts. | Use skills for bounded workflows once repetition and acceptance criteria are observed—for example, a source-grounded dossier workflow—not for vague “help with design.” Keep deterministic transforms/validators in reviewed scripts called by the workflow. |
| **3. Policy skills** | Policies require continuous authority; skills load conditionally. A policy skill can explain or audit a policy, but cannot be the only carrier of witness, sacred-event, source-separation, or no-invention requirements. | Keep enduring constraints in the Master Protocol and future verified repository-instruction layer; later add schema/runtime/permission enforcement and evaluations where possible. A policy-audit skill may compose with those sources but must cite rather than duplicate them as authority. |
| **4. Domain-reference skills** | On-demand reference packaging can reduce irrelevant context and pair vocabulary with use instructions. However, reference authority depends on provenance, edition, confidence, and freshness, not `SKILL.md`. Large or changing corpora are poor candidates for blind duplication. | Keep canonical Scripture, research, chronology, and source records as separately governed domain artifacts. A skill may route to the smallest relevant sources and enforce citation/confidence handling. Do not embed unverified OCR as “knowledge.” |
| **5. Tool-operation skills** | A skill can teach safe operation of an engine, DCC tool, build system, MCP tool, or script. It neither installs the tool nor grants permission. Tool-specific guidance is premature while engine, DCC, build, schema, and validation choices remain unknown. | After stack selection, create focused operating workflows around verified tools and versions. Keep credentials, MCP configuration, scripts, and permission rules in their native artifact classes; let the skill state prerequisites and verification steps. |
| **6. Orchestrator/router skills** | Routing selects roles, skills, agents, tools, or approval paths. It may be a skill, agent instruction, application/router configuration, or ordinary main-agent judgment. Premature routers fossilize a taxonomy and amplify ambiguous triggers. | Defer a router until repeated tasks reveal stable boundaries and a tested small catalog exists. Preserve explicit user/Creative Director routing and conflict escalation. If later added, route by task/output/risk, not impressive job titles. |
| **7. Artifact-production skills** | Research dossiers, quests, dialogue, code, codex entries, visual briefs, schemas, and validation reports have different authority and acceptance needs. A repeatable production workflow may suit a skill, but production artifacts themselves are outputs, not skills. | Define each output contract, sources, approver, and evaluation first. Schema/content generation must wait for an approved schema or an explicit schema-design task. Compose production workflows with role, domain, tool, and validation artifacts rather than building one monolithic producer. |

## 6. Common category errors

1. **Equating a role with an agent.** “Scripture & Theology Researcher” currently assigns responsibility; it does not prove a separately configured or spawned runtime.
2. **Equating a prompt with authority.** Repeating “must” in an activation prompt does not outrank the Master Protocol, current user decisions, source commitments, or execution-environment instructions.
3. **Using a skill as the sole policy store.** Conditional loading can omit the very rule intended to be universal.
4. **Treating references as instructions or truth.** A dossier can inform work without governing it; an OCR corpus can contain text without being a verified edition.
5. **Treating a skill as a tool grant.** Instructions to call an engine, MCP server, or build command do not make that capability installed, authenticated, safe, or permitted.
6. **Treating an MCP server as one tool.** A server may expose multiple tools, resources, and prompts with separate risk and approval profiles.
7. **Treating scripts as harmless attachments.** Scripts can change state and require code review, bounded inputs, permissions, and tests independent of the prose skill.
8. **Treating a guardrail as enforcement.** A written sacred-event rule is necessary governance, but runtime invariants and adversarial tests would be distinct future controls.
9. **Treating handoff or memory as a decision log.** Summaries can be stale or wrong; durable decisions need provenance, confirmation, and supersession rules.
10. **Treating an evaluation as a benchmark—or a passing benchmark as approval.** Narrow cases can miss theological, historical, aesthetic, security, or gameplay failures.
11. **Encoding unknown stack choices into reusable artifacts.** Unity, C#, editor commands, build steps, content paths, and MCP tools remain unknown; a skill that assumes them would convert absence into false authority.
12. **Creating a universal “Bible expert” skill.** It would collapse Scripture, textual variants, interpretation, historical reconstruction, tradition, and creative proposal—the exact distinctions the project must preserve.

## 7. Implications for The Bible Video Game

### Findings

- The project’s non-negotiable identity is not a persona choice: the player is a witness, sacred events are fixed and protected, meaningful agency surrounds rather than rewrites them, and Scripture/reconstruction/invention must remain distinct. These are governing project constraints in the [Master Protocol](https://github.com/Degrading1919/The_Bible_Video_Game/blob/e121e44c209d496ad0a5ac5b61bd402fb5e2bd62/00_ADMIN/Reference/The_Bible_Video_Game_Master_Assistant_Protocol.md).
- The present role architecture is coherent as manual governance. It does not require immediate conversion into nine skills or nine agents.
- The project’s Scripture database/codex vision makes provenance, confidence, era-appropriate knowledge, canon category, and player-facing versus in-world knowledge first-class future data concerns. Those concerns span policy, domain reference, schema, workflow, and evaluation; no single skill class contains them.
- The repository remains a documentation/research foundation. Engine, language, schemas, content contracts, DCC tools, build/test workflow, MCP services, hooks, and evaluation harness are unknown or absent. Session 2 taxonomy cannot select them by implication.
- Sacred-event protection currently exists as design policy. Technical guardrails, protected-entity invariants, interaction restrictions, failure behavior, and exploit tests remain future implementation questions.

### Recommendations

1. Preserve a **source-of-authority spine**: user/Creative Director decisions, core witness vision, biblical source commitments, approved scope/canon decisions, Master Protocol, then approved schemas/specs and task instructions. Reusable artifacts should cite this spine rather than clone it.
2. Classify every future reusable artifact before building it: owner, authority, scope, trigger, inputs, outputs, action permissions, cargo, failure modes, and evaluation. “Make it a skill” should be a conclusion, not a starting assumption.
3. Favor skills first where the project already has stable repeatable procedure: source selection, confidence labeling, separation of fact/reconstruction/proposal, artifact formatting, and validation handoff. Keep role ownership and universal policy outside conditional activation.
4. Require every future artifact-production workflow to preserve witness constraints and explicitly identify sacred-event risk, era knowledge, source category, uncertainty, and human approval points.
5. Delay engine/DCC/build/MCP operating artifacts until the user selects the stack. Then bind instructions to exact supported versions and keep executable capability separate from prose guidance.
6. Build evaluations alongside any later skill: positive cases, boundary cases, and negative cases such as invented certainty, Westernized assumptions, player replacement of sacred figures, schema invention, anachronistic knowledge, and canonical/non-canonical blending.
7. Keep final theological, canon, historical-confidence, sacred-figure, and major creative approvals with accountable humans. An agent may research, flag, and recommend; neither a role label nor a benchmark score transfers that stewardship.

## 8. Terms that remain ambiguous or vendor-dependent

| Term | Unresolved/vendor-dependent dimension | Required discipline in later sessions |
|---|---|---|
| **Agent / subagent** | Context inheritance, isolation, nesting, lifecycle, model choice, and permission inheritance differ across Codex, Copilot, and Claude. | Name the exact product, surface, and version before making capability claims. |
| **Skill** | Common `SKILL.md` core exists, but discovery paths, invocation controls, metadata extensions, tool allowances, and packaging vary. | Separate open-standard fields from Codex-native behavior and third-party extensions. |
| **Command** | May mean shell execution, client control, slash-command UI, prompt expansion, or registered tool. | State interpreter/host and whether the command causes effects or sends instructions. |
| **Memory** | Storage, retention, provenance, visibility, editability, and sharing vary widely. | Do not claim memory until all six are specified; prefer versioned repo records for approved project truth. |
| **Policy / rule / guardrail** | Vendors use “rules” for different layers, from prose guidance to command authorization. | State whether the control is advisory, detective, approval-gated, or technically blocking. |
| **Prompt / system prompt / developer instruction** | Message roles and override behavior are platform-specific. | Use vendor terminology only for artifacts actually placed at that layer. |
| **Agent configuration** | File format and supported model/tool/MCP/sandbox fields change by ecosystem and may evolve. | Treat role content as portable source material, not portable configuration. |
| **Evaluation / benchmark** | Automated versus human scoring, deterministic versus model-graded results, and acceptable thresholds are project choices. | Define evaluator, rubric, dataset provenance, version, repeatability, and approval consequence. |
| **Hook** | Event names, ordering, blocking behavior, error handling, and permission context vary. | Verify exact host semantics after the development stack is selected. |
| **Native** | A concept may be native to a Codex surface, supported via an open standard, or merely usable through generic files/tools. | Use “Codex-native,” “cross-ecosystem standard,” “vendor-specific,” and “generic” precisely rather than calling everything compatible. |
